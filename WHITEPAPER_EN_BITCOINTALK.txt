[size=16pt][b]1. Preface: History and Meaning[/b][/size]

[size=12pt][b]Project Origins[/b][/size]

[b]Ninacatcoin[/b] was born from the [b]Monero[/b] codebase, one of the most respected cryptocurrency projects in terms of privacy and decentralization. Just as Monero was based on the CryptoNote protocol, Ninacatcoin uses Monero's solid, battle-tested architecture as its foundation, implementing fundamental improvements in several critical areas:

[list]
[li][b]Anti-51% security:[/b] Automatic checkpoints every 60 minutes vs manual checkpoints updated every 6–12 months in Monero[/li]
[li][b]Mining fairness:[/b] Enhanced RandomX with GPU penalty to prevent mining monopolies[/li]
[li][b]Transparent emissions:[/b] X2 and X200 high-entropy events with anti-manipulation mechanisms[/li]
[li][b]Real decentralization:[/b] No manual intervention required in critical processes[/li]
[li][b]Rapid difficulty adjustment:[/b] LWMA-1 algorithm reacts within blocks, not days or weeks[/li]
[/list]

This project respects and acknowledges the work of the Monero community as a technical foundation, while implementing an alternative vision of how a private cryptocurrency should operate. Monero represents the gold standard in privacy-focused cryptocurrencies, but several design choices — particularly around manual checkpoints, difficulty adjustment speed, and mining reward predictability — left room for meaningful improvement. Ninacatcoin fills these gaps while preserving privacy, fungibility, and resistance to surveillance.

[size=16pt][b]5. High-Entropy Events: X2 and X200[/b][/size]

[size=12pt][b]What Are X2 and X200?[/b][/size]

Ninacatcoin implements [b]pseudorandom events[/b] that multiply the block reward at unpredictable intervals:

[list]
[li][b]X2:[/b] Block pays 2x the base reward (e.g., 8 NINA in year 1)[/li]
[li][b]X200:[/b] Block pays 200x the base reward (e.g., 800 NINA in year 1)[/li]
[/list]

[b]Expected frequency per year (~262,800 blocks):[/b]
[list]
[li]X2: ~183 events/year (one every ~1,436 blocks, roughly every 2 days)[/li]
[li]X200: ~6 events/year (one every ~43,800 blocks, roughly every 2 months)[/li]
[/list]

These events are [b]deterministic but unpredictable[/b] — every node calculates the same result, but no one can predict it before the previous block is mined.

[size=12pt][b]Probability Calculation and Anti-Manipulation[/b][/size]

Event detection is [b]entirely deterministic based on the previous block hash[/b], never on unseeded random numbers, system clocks, or external oracles:

[code]
auto get_event_roll = [&](uint8_t tag, uint64_t &out) {
    if (prev_block_hash) {
        std::array<uint8_t, sizeof(crypto::hash) + 1> buf{};
        std::memcpy(buf.data(), prev_block_hash, sizeof(crypto::hash));
        buf[sizeof(crypto::hash)] = tag;
        const crypto::hash h = crypto::cn_fast_hash(buf.data(), buf.size());
        std::memcpy(&out, &h, sizeof(uint64_t));
    }
};
[/code]

[b]Process:[/b]
[list]
[li]Take the previous block hash (32 bytes, 256 bits)[/li]
[li]Append an identifier tag (1 byte): tag=2 for X2, tag=200 for X200[/li]
[li]Apply Keccak-256 (cn_fast_hash) to the combined 33-byte buffer[/li]
[li]Extract the first 64 bits as an unsigned integer[/li]
[li]Calculate: rnd % BLOCKS_PER_YEAR (262,800)[/li]
[li]If result < 183 → X2 event; If result < 6 → X200 event[/li]
[/list]

[b]Why manipulation-resistant?[/b] Changing the block hash requires re-mining (full PoW), the miner doesn't know if the next block will be theirs, even with 100% hashrate the cost exceeds the expected reward, and Keccak-256 has no shortcuts.

[size=12pt][b]Emission Limits[/b][/size]

[list]
[li][b]Supply limit:[/b] If already_generated_coins + (base_reward × multiplier) > MONEY_SUPPLY, the reward is reduced to exactly what remains[/li]
[li][b]Hard brake:[/b] If fewer than 10,000 NINA remain, X2/X200 are completely disabled[/li]
[/list]

Block at height 100 is [b]always X2[/b], demonstrating the system's functionality on the live network.

[size=16pt][b]6. Difficulty Algorithm: LWMA-1[/b][/size]

[size=12pt][b]Target Block Time[/b][/size]

Ninacatcoin targets a block time of [b]120 seconds[/b] (DIFFICULTY_TARGET_V2 = 120), producing approximately 30 blocks per hour, 720 blocks per day.

[size=12pt][b]The Problem with Simple Moving Average (SMA)[/b][/size]

The original Monero difficulty algorithm uses SMA over 720 blocks (~24 hours). It has severe limitations for smaller networks:

[list]
[li][b]Slow reaction:[/b] 50% hashrate loss takes ~24 hours to adjust[/li]
[li][b]Equal weighting:[/b] Old blocks influence difficulty as much as new ones[/li]
[li][b]Timestamp sorting vulnerability:[/b] Manipulation possible[/li]
[li][b]Hash-and-dump susceptibility:[/b] Network takes a full day to recover[/li]
[/list]

In January 2025, Ninacatcoin experienced exactly this: a miner with ~210 KH/s departed, leaving ~1 KH/s against difficulty calibrated for 215 KH/s. With the old SMA, the network would have needed ~29 days to normalize.

[size=12pt][b]LWMA-1: The Solution[/b][/size]

Ninacatcoin uses the [b]LWMA-1[/b] algorithm (Linear Weighted Moving Average), designed by zawy12 — the industry standard for difficulty adjustment in small/medium cryptocurrencies.

[b]Key parameters:[/b]
[code]
DIFFICULTY_WINDOW  = 60 blocks (~2 hours of data)
DIFFICULTY_LAG     = 2  blocks
DIFFICULTY_CUT     = 6  timestamps (excluded from extremes)
[/code]

[b]Core Formula:[/b]
[code]
next_difficulty = (sum_difficulty × T × (N + 1)) / (2 × sum_weighted_solvetimes)

Where:
  T   = target time (120 seconds)
  N   = number of intervals in window (up to 60)
  D_i = difficulty of block i
  st_i = solve time of block i, clamped to [-6T, 6T]
  weight_i = i (1 for oldest, N for newest)
[/code]

[b]LWMA-1 vs SMA comparison:[/b]

[table]
[tr][td][b]Feature[/b][/td][td][b]SMA (Monero)[/b][/td][td][b]LWMA-1 (Ninacatcoin)[/b][/td][/tr]
[tr][td]Window size[/td][td]720 blocks (~24h)[/td][td]60 blocks (~2h)[/td][/tr]
[tr][td]Weighting[/td][td]Equal for all blocks[/td][td]Linear: recent weigh more[/td][/tr]
[tr][td]Timestamp handling[/td][td]Sorted (loses order)[/td][td]Unsorted (real solve times)[/td][/tr]
[tr][td]Reaction speed[/td][td]Hundreds of blocks[/td][td]3–5 blocks[/td][/tr]
[tr][td]Timestamp manipulation[/td][td]Vulnerable[/td][td]Clamped to [-6T, 6T][/td][/tr]
[tr][td]99% hashrate loss recovery[/td][td]~29 days[/td][td]1–2 blocks (with EDA)[/td][/tr]
[/table]

The most recent block contributes [b]60x more[/b] to the difficulty calculation than the oldest block, allowing the algorithm to react within 3–5 blocks (~6–10 minutes).

[size=12pt][b]Emergency Difficulty Adjustment (EDA)[/b][/size]

If a block takes more than 6x the target time (>720 seconds), EDA overrides LWMA and caps difficulty:

[code]
EDA Trigger: last_block_time > 6 × 120s = 720s (12 minutes)

adjusted_difficulty = last_difficulty × target / actual_solve_time
next_difficulty = min(LWMA_result, adjusted_difficulty)
[/code]

[b]Example — catastrophic hashrate departure:[/b]
Difficulty 9,000,000 (calibrated for 200 KH/s), hashrate drops to 1 KH/s, block takes 3,600s (1 hour):

[code]
adjusted = 9,000,000 × 120 / 3,600 = 300,000
→ Next block mined at difficulty 300,000 instead of 9,000,000
[/code]

[size=12pt][b]Difficulty Reset Mechanism[/b][/size]

At height 4,726, a one-time difficulty reset occurred (DIFFICULTY_RESET_HEIGHT = 4726). The old SMA had 720 blocks of history at ~215 KH/s, while actual hashrate was ~1 KH/s. The reset prevented LWMA from looking back before the reset height. Difficulty dropped from 9.15M to ~68K within 60 blocks. Future fluctuations are handled entirely by LWMA-1 + EDA.

[size=12pt][b]Solve Time Clamping[/b][/size]

LWMA-1 clamps each solve time to [-6T, 6T] = [-720s, +720s], preventing future timestamp attacks, slow block exaggeration, and oscillation attacks.

[size=16pt][b]8. Privacy and Anonymity[/b][/size]

[size=12pt][b]Ring Signatures[/b][/size]

Every transaction uses [b]mandatory ring signatures[/b] — enforced at protocol level. The sender's real output is mixed with N-1 decoy outputs:

[list]
[li]Default ring size: [b]11[/b] (mixin 10) — 1 real + 10 decoys[/li]
[li]Probability of guessing the real signer: ~9.1%[/li]
[li]Future hard fork (v17) will increase to ring size [b]16[/b] once the blockchain matures[/li]
[/list]

Ring size 11 was chosen for the initial phase because a young blockchain has limited outputs available. Starting at 11 ensures strong privacy at every growth stage.

[size=12pt][b]RingCT (Ring Confidential Transactions)[/b][/size]

[list]
[li][b]Pedersen Commitments:[/b] Amounts encrypted: C = aG + bH[/li]
[li][b]Homomorphic property:[/b] Miners verify inputs = outputs without knowing amounts[/li]
[li][b]Bulletproof+ range proofs:[/b] Compact proofs that amounts are non-negative[/li]
[li]Enforced since hard fork v6[/li]
[/list]

[size=12pt][b]Stealth Addresses[/b][/size]

Every transaction generates a unique, one-time address for the recipient. No two transactions ever send to the same on-chain address. An observer cannot link payments to a known public address.

[size=12pt][b]Tor and I2P Integration[/b][/size]

[list]
[li]Nodes detect .onion and .b32.i2p addresses as anonymous endpoints[/li]
[li][b]Dandelion++[/b] provides transaction propagation privacy: Stem phase (single peer forward) → Fluff phase (network broadcast)[/li]
[li]Impossible to determine the originating node[/li]
[/list]

[tr][td]Sender identity[/td][td]Yes[/td][td]Ring signatures (11 decoys)[/td][/tr]

[size=16pt][b]9. Enhanced RandomX Mining[/b][/size]

RandomX is activated at block height 3,000 (hard fork v16). Blocks 0–2,999 used CryptoNight. Ninacatcoin implements three improvements over standard RandomX:

[size=12pt][b]Improvement 1: Shorter Epochs[/b][/size]

[code]
Monero:       SEEDHASH_EPOCH_BLOCKS = 2048 (68 hours)
Ninacatcoin:  SEEDHASH_EPOCH_BLOCKS = 1024 (34 hours)
[/code]

More frequent seed changes make ASIC ROI mathematically impossible (design takes 6–12 months, seed changes every 34 hours).

[size=12pt][b]Improvement 2: Variable Dataset[/b][/size]

[code]
dataset_size = min(2GB + (network_hashrate_TH_s × 10MB), 4GB)
[/code]

ASICs optimized for 2GB become suboptimal when the dataset grows. CPUs handle any dataset size transparently.

[size=12pt][b]Improvement 3: Dual Mode with GPU Penalty[/b][/size]

Every 5 blocks (20%), RandomX runs in "safe" mode (RANDOMX_FLAG_SECURE) with JIT disabled:

[code]
Block 0 (height % 5 == 0): GPU Penalty Mode (10x slower for GPUs)
Blocks 1–4: Normal mode
Average GPU efficiency: ~20% vs CPU baseline
[/code]

[size=12pt][b]CPU Mining: Democratic Access[/b][/size]

RandomX uses only basic CPU instructions (integer arithmetic, L3 cache, memory latency). A used office computer from 2010 can mine profitably.

[size=16pt][b]10. Automatic Checkpoints: The Impossible Defense Against 51%[/b][/size]

[size=12pt][b]Why Checkpoints Matter[/b][/size]

A 51% attack rewrites blockchain history using majority hashrate. Checkpoints make it [b]cryptographically impossible[/b] to rewrite history, even with 100% of the hash power.

[size=12pt][b]Checkpoint Architecture[/b][/size]

A checkpoint is a pair (height, block_hash) that permanently locks a block. Once validated against a checkpoint, a block CANNOT be replaced.

[list]
[li][b]Checkpoint interval:[/b] 30 blocks = 3,600 seconds = 1 hour[/li]
[li][b]Distribution:[/b] HTTPS + seed nodes[/li]
[li][b]Generation:[/b] Fully automatic, no human involvement[/li]
[/list]

[size=12pt][b]Why 51% and 100% Attacks Fail[/b][/size]

[b]Formal proof:[/b] For the attack to succeed, the alternative chain R_A must match checkpoints. But checkpoints were generated from the honest chain R_H. So R_A = R_H at every checkpoint. Since blocks are chained, R_A = R_H everywhere. But then R_A IS R_H — not an alternative chain. Contradiction. QED.
